## Product

<!-- /product appends entries here when it learns something during intake, classification, or close-out that should inform future tasks. -->

- **Not every OAuth-gated feature has a server-side verification workaround — some are blocked by the sign-in popup itself, before any of the app's own code runs.** For the bust-pause/animation task, the earlier trick of exercising server functions directly (bypassing HTTP/auth) didn't apply, because the feature is pure client-side presentation reacting to game state that can only be reached by actually playing through a live, real-account session — there's no "just call the function" shortcut. Worth distinguishing at intake time between "OAuth-gated but the logic is independently testable" (most of this project's dice-game backend) and "OAuth-gated with no bypass at all" (client-only presentation features) — the latter should be flagged to the user as needing a manual pass from the start, not discovered as a surprise at review time.
- **When a user's manual playtest surfaces something that looks like a bug, check whether the responsible code was actually touched by the task before treating it as a regression.** After the AI-opponent task, the user asked if the AI winning 103–99 against a winning score of 100 was a bug (overshooting the target). It wasn't — `hold()`'s `bankedScore >= winningScore` win condition in `src/lib/dice-game/engine.ts` predates the task entirely and applies identically to human players; nothing in the AI-opponent diff touched `engine.ts`. A one-line grep/diff check settled it immediately. Worth doing that check before assuming a live-tested feature introduced new behavior — the "bug" is often just newly-visible pre-existing behavior.

- **For a bug report that only describes a symptom (a screenshot, "X shows the wrong thing"), do a quick root-cause grep before writing the task file, not after handoff.** For the "opponent shows raw UID instead of name" bug, tracing `displayName` through `PlayerBadge` → `GameBoard` → `serialize.ts` → `playerStore.ts`/`verifyRequest.ts` during intake made it possible to write a `## Problem` section naming the exact root cause and file:line, which let the Requirements section state precisely what needed to change instead of a vague "fix the display bug." Costs a few minutes of intake time; pays for itself in a much tighter developer handoff and a verification pass that can check specific files instead of guessing.
- **A backlog item phrased as "persist X (e.g. database or local storage)" hides several independent decisions — ask them all before writing the task file, not just the obvious one.** For "persist game/player data," the real questions turned out to be (1) what survives (browser refresh vs. server restart vs. shared-across-devices — very different tech: localStorage vs. a real DB), (2) *which* data specifically (win tally vs. active game vs. player profiles — picking all three would have been a much bigger task than the user actually wanted), and (3) which backend, where the obvious guess (Firestore, since Firebase auth is already wired up) was wrong — a `MONGO_URI` was already sitting provisioned and unused in `.env.local`, discovered only by checking. Getting all three answered up front turned a vague, potentially sprawling task into a single tightly-scoped one. When a backlog item names a category of solution ("a database") rather than a specific one, treat that as a sign multiple decisions are bundled together — surface each one as its own question.
- **When a stated requirement's literal edge case would require new infrastructure beyond the task's actual scope (e.g. a new endpoint just to prefetch a value for a screen that doesn't otherwise need one), it's fine for the developer to implement the core ask and flag the narrower interpretation explicitly at close-out rather than blocking mid-implementation or silently guessing.** For "track player wins," the requirement said the count should be "visible throughout" — the developer covered every screen with real game data but skipped the one-time pre-game setup screen (which has no server round-trip yet), and surfaced that choice plainly in Implementation Notes. Asking the user a single yes/no at review time ("leave as is" or "cover it too") resolved it in one round-trip instead of a back-and-forth mid-task. Reasonable default when scoping any requirement with an ambiguous edge case: build the clearly-in-scope core, name the gap, let the user make the final call at verification.
- **Auth/OAuth-heavy goals hit a hard verification ceiling in this environment.** Across the dice-game goal (Google Sign-In backend, API, sign-in UI, game board), every task's real interactive path — an actual Google OAuth popup, a real ID token reaching the backend — was un-testable without a human in a real browser with real accounts. Code-level verification (typecheck, build, unit tests on pure logic, live error-path checks against the running server) can go far, but the happy path always needs a manual pass at the end. When scoping a similar goal, plan for that manual checkpoint explicitly rather than treating "all tasks verified" as equivalent to "actually works" — and prefer bundling that manual test at the *end* of the goal (once the full stack is wired) rather than per-task, since intermediate tasks (e.g. a bare API route) often have nothing a human can click through yet.

## Design

<!-- /designer appends entries here: tokens that turned out wrong, patterns that worked well, recurring brief gaps. -->

- **A feature that sounds new ("AI opponent") can still resolve to zero new tokens/patterns if the brief starts from the existing inventory rather than the feature name.** For the AI opponent's "Play vs AI" button and "AI is thinking…" indicator, every visual need was already covered: the Secondary outline button pattern (identical to the existing sign-in button), the Loading spinner + status-role pattern (already used twice), and `PlayerBadge`'s initials fallback (no new avatar treatment for a player with no photo). Emoji-as-icon (🤖) followed the precedent already set by the 🏆 wins pill. Before reaching for a new pattern, check whether the feature is actually a new *composition* of existing ones — most "new-sounding" features in a maturing design system are.

- **Partial rebrand scoped to "tokens/patterns only, don't restyle existing pages" needs a namespaced parallel token set, not edits to the existing tokens.** For the IBI brand refresh, added `--color-ibi-*` alongside the existing `--color-*` tokens rather than changing existing values — editing `--color-accent` etc. directly would have restyled every existing component as a side effect, which the task explicitly ruled out. Same logic applied to the button: a new, separately-named `IbiSecondaryButton` component rather than modifying the existing "Secondary (outline) button" pattern in place. This is the pattern to reach for any time a brief says "introduce new visual language but don't touch what's already built" — namespace everything new, migrate specific components later as their own deliberate tasks.

## Development

- **CSS transform breaks position:fixed on children** — any parent with `transform` (including hover animations like `translateY`) creates a new stacking context, making `position: fixed` descendants scroll with the page instead of the viewport. Fix: render modals/overlays via `ReactDOM.createPortal(..., document.body)`. Guard with a `mounted` state to avoid SSR issues in Next.js App Router.
- **Firebase config in `.env.local` isn't `NEXT_PUBLIC_`-prefixed** (`API_KEY`, `AUTH_DOMAIN`, `PROJECT_ID`, `STORAGE_BUCKET`, `MESSAGING_SENDER_ID`, `APP_ID`) — fine for server-side use (e.g. verifying tokens), but any task that needs the Firebase **client** SDK in the browser will need those values exposed some other way (rename with `NEXT_PUBLIC_` prefix, or serve via a small API route) since unprefixed env vars aren't in the client bundle.
- **Verifying Firebase/Google ID tokens without a service account**: no `FIREBASE_*` service-account credentials exist in this project, so `firebase-admin` can't initialize normally. Instead, verify the JWT directly with `jose`'s `createRemoteJWKSet` against Google's public JWKS (`https://www.googleapis.com/service_accounts/v1/jwk/securetoken@system.gserviceaccount.com`), checking `issuer: https://securetoken.google.com/<PROJECT_ID>` and `audience: PROJECT_ID`. Implemented in `src/lib/auth/verifyRequest.ts`.
- **Two simultaneous Firebase-authenticated users in one browser tab need two named Firebase apps.** The Firebase JS Auth SDK holds exactly one active session per `Auth` instance — signing in a second Google account on the default `Auth()` silently signs the first one back out. Fix: `initializeApp(config, "player1")` / `initializeApp(config, "player2")`, each with its own `getAuth(app)`. Guard against re-initialization on re-render with `getApps().find(app => app.name === slot)` before calling `initializeApp` again. Implemented in `src/lib/firebase/client.ts`. Also: Firebase ID tokens expire (~1hr) — retain the live `User` object (or a bound `getIdToken` closure) per session rather than a captured token string, so callers can always fetch a fresh token instead of hitting a stale one later.
- **`_`-prefixed folders under `src/app/` are invisible to Next.js App Router routing** — they're treated as private/co-location folders and excluded from route generation, so a `page.tsx` inside one silently 404s. If you ever need a throwaway route for manual verification of a component nothing else uses yet, don't prefix it with `_` — use a plain name, verify, then delete the whole folder afterward. Also: deleting a route folder can leave a stale reference in `.next`'s generated dev types (harmless — `rm -rf .next` clears it).
- **`src/app/favicon.ico` is a separate, file-convention favicon source from `metadata.icons` in `layout.tsx`** — Next.js App Router auto-detects and serves it at `/favicon.ico`, independently of whatever `metadata.icons` declares. Changing `metadata.icons` alone is not enough to change "the favicon" as a user experiences it; if `src/app/favicon.ico` exists, it must be updated (or removed) too, or the browser tab will keep showing the stale file regardless of the metadata config. Caught this during the IBI favicon swap — the file predated even the previous placeholder set and would have silently kept serving.
- **This project has a `MONGO_URI` already provisioned in `.env.local` (no driver installed, nothing referencing it) — it's the intended persistence backend, not Firestore, even though Firebase is already used for auth.** Discovered while scoping "persist player wins"; worth checking `.env.local` for unused connection strings before assuming a new persistence task should reach for whatever's already partially wired up (Firebase here). Implementation pattern used: `src/lib/db/mongo.ts` caches the `MongoClient` connect promise on `globalThis` (survives dev hot-reload and warm invocations, avoids exhausting connection limits by reconnecting per call) — reuse this helper for any future MongoDB-backed feature rather than opening a new client per module.
- **When a feature needs live verification but the only real trigger is behind Google OAuth (un-driveable headlessly, per the existing Product-section note), and the change is a storage-backend swap, verify by hitting the storage directly with the same queries the code uses, rather than skipping verification entirely.** For "persist player wins," ran the exact same MongoDB collection name, database name, and atomic `findOneAndUpdate($inc, upsert)` operation the new `winStore.ts` uses, confirmed the increment and a fresh read both work, then deleted the test document. Gives real confidence in the persistence logic without needing a browser session — reserve this technique for backend-only changes where the verification gap is specifically "no way to trigger the real request," not "the logic itself is unverified."
- **`src/lib/dice-game/session.ts`'s `applyEngineResult` is the single chokepoint both `rollForPlayer` and `holdForPlayer` funnel through to merge a new `GameState` into `activeSession`** — it's the right place to hook any side effect that should fire exactly once on a state *transition* (e.g. incrementing a win tally when `status` flips from `"in_progress"` to `"finished"`), by comparing the old and new state there rather than re-deriving "did something just happen" from the new state alone (which would double-fire on every re-serialization of an already-finished game). Note `startGame`/`resetGame` bypass `applyEngineResult` entirely (they construct a fresh state directly), so a hook placed there only fires for in-game actions, not game creation/reset — that was the desired behavior for win tracking, but worth remembering if a future transition-based side effect *does* need to also fire on reset.
- **Only the request's own bearer token gets verified/registered — a second player's uid passed as a bare string in a request body does NOT get their profile into `playerStore`.** `POST /api/game` (game creation) only ran `authenticateRequest` (verifies the `Authorization` header) for the caller; `player2Uid` arrived unauthenticated, so `getPlayer(player2Uid)` returned undefined until player2 personally made an authenticated call (their first roll/hold) — until then their badge showed the raw uid instead of a name. Fixed by extracting the token-verify-and-upsert logic out of `authenticateRequest` into a standalone `verifyIdToken(token)` (`src/lib/auth/verifyRequest.ts`, exported from `src/lib/auth/index.ts`), then having the client send the *other* signed-in player's ID token too (`player2IdToken` in the body) since, per the two-named-Firebase-apps pattern, the client already holds both tokens locally — the server just wasn't asking for the second one. Reusable pattern: any endpoint that accepts a second/other party's uid as a plain string should also accept and verify that party's own token if the goal is to have their real profile data available immediately, rather than waiting for them to separately authenticate.
- **A client "use client" module that imports `@/lib/dice-game` (the barrel) would bundle `mongodb` into the browser** — the barrel re-exports `session.ts` → `winStore.ts` → `src/lib/db/mongo.ts`, which imports the Node-only `mongodb` package. When the AI opponent needed a shared identity constant (`AI_PLAYER_UID`) in both server code (API routes) and client components (`PlayerSignIn.tsx`, `GameBoard.tsx`), the fix was a dedicated leaf module, `src/lib/dice-game/aiPlayer.ts`, with **zero imports of its own** — client code imports it directly (`@/lib/dice-game/aiPlayer`), never through the barrel. Rule of thumb for this codebase: anything that needs to be shared between a "use client" component and server-only game/db logic must live in its own dependency-free leaf module; only server code (API routes) should import the `@/lib/dice-game` barrel.
- **A synthetic/non-authenticated "player" (no real Firebase account) breaks any call site that unconditionally calls `.getIdToken()` on both players** — for the AI opponent, `GameBoard.tsx`'s `handleCreate` (fetched both tokens via `Promise.all` unconditionally) and `handleReset` (via `resolveActingPlayer`, which could resolve to whichever player matches the current turn) both needed fixing once `player2` could be a non-real `AuthedPlayer`. Made `createAiPlayer()`'s `getIdToken` throw explicitly rather than return a dummy value, specifically so any missed call site fails loudly during testing instead of silently sending garbage to the server. Before adding any second "virtual" player type to this codebase, grep every `.getIdToken()` call site first — don't assume the existing two-human call sites are safe defaults.
- **`src/app/page.tsx` renders `PlayerSignIn` OR `GameBoard`, never both — a "should be visible on every screen" control placed inside `GameBoard` misses the sign-in screen entirely**, since `GameBoard` doesn't mount until after `page.tsx`'s own `players` state is set. Caught this while wiring up the sound-effects mute toggle: the Design Brief only reasoned about `GameBoard`'s own two internal phases (pre-game setup vs. active board) and didn't account for the screen `page.tsx` shows *before* `GameBoard` exists at all. Fixed by hoisting the control to `page.tsx` itself (rendered once, alongside whichever child is currently active) and switching its CSS from `position: absolute` (needs a specific positioned ancestor, tying it to wherever it's rendered) to `position: fixed` (viewport-anchored, works from any mount point). When a brief says a control should be "always visible" or "regardless of phase," check the *actual* top-level render tree (`page.tsx`'s conditionals), not just the component being modified — the full screen sequence here is sign-in → pre-game setup → active board, three states across two components, not two states within one.
- **A prior cleanup task's "KEPT — used in `file:line`" citation is a snapshot, not a permanent guarantee — re-verify with a fresh grep rather than trusting the old note.** `remove-unused-components.done.md` kept `SkillCard` because `src/app/skills/page.tsx` used it; that whole route was deleted afterward (during a homepage cleanup step), leaving `SkillCard` orphaned again with no re-audit triggered. Caught only because this later prune task re-ran the grep from scratch instead of trusting the earlier verdict. When re-auditing "unused component" candidates, always grep current state — don't shortcut past ones a previous task already cleared as "used."
- **Two `useEffect`s that both depend on the same state value cannot coordinate through that state within a single commit** — each effect's closure only sees the value from the render that scheduled it, not any update a sibling effect made moments earlier in the same flush. When the bust-pause feature needed to guarantee "bust pause, then the AI's separate thinking pause, never in parallel," a second independent `useEffect` (alongside the existing AI-turn one, both keyed on `[gameState]`) would have raced: the AI-turn effect would see the stale pre-bust `pendingAction` and start its own timer concurrently. Fix: merged both into one `useEffect` with a single async function that `await`s the first delay before ever checking the condition for the second — sequencing becomes structural (a `then`), not a hope that two independent timers happen to not overlap. Reach for one effect with an internal `await` sequence, not two effects coordinating via shared state, whenever two delayed steps must never run concurrently.
